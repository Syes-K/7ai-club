# 用户故事：多会话、SQLite 与精简请求（version=0.0.2）

## US-1：新建对话

**作为** 对话用户，**我想要** 新建一条独立的对话会话，**以便** 在不混杂旧主题的情况下开始提问。

### 验收标准（AC）
- [ ] 界面提供明确入口（如「新对话」按钮），点击后产生**新的** `sessionId`（或等价标识），消息区为该会话的空或初始状态。
- [ ] 新建后发送的第一条消息仅归属新会话，不出现在此前会话列表展示的内容中。
- [ ] 新会话在持久化存储中有对应记录（会话表或等价结构可查）。

---

## US-2：查看并切换历史会话

**作为** 对话用户，**我想要** 看到已有会话列表并切换到某一历史会话，**以便** 继续之前的讨论。

### 验收标准（AC）
- [ ] 列表（或等价 UI）展示至少包含可区分项：默认标题可用首条用户消息截断、时间或「未命名会话」等（具体文案由设计定）。
- [ ] 点击某一历史项后，主区域展示该会话的**已存消息顺序**，与存储一致。
- [ ] 切换会话后，下一次 API 调用携带的 **sessionId** 与当前选中会话一致。
- [ ] 快速连续切换时不会出现 A 会话消息显示在 B 会话下的串线（竞态需处理或定义加载态）。

---

## US-3：刷新或重新打开后历史仍在

**作为** 对话用户，**我想要** 关闭标签页或刷新后仍能看到之前的会话与消息，**以便** 依赖本地持久化继续工作。

### 验收标准（AC）
- [ ] 使用 SQLite 持久化后，在同机同数据文件前提下，刷新页面后会话列表与消息内容可恢复。
- [ ] 未显式删除的会话在重启开发服务后仍存在（在同一 DB 文件路径下）。

---

## US-4：服务端按会话加载上下文

**作为** 系统，**我想要** 在收到聊天请求时根据 **sessionId** 从存储读取该会话历史并组装模型上下文，**以便** 客户端不必上传全量历史。

### 验收标准（AC）
- [ ] 合法请求在包含 `sessionId` 时，模型调用所使用的 `messages`（或等价结构）由**服务端**基于存储记录与本轮用户输入组装。
- [ ] 对不存在的 `sessionId` 返回明确错误（4xx）或按产品约定自动创建会话（任选一种并在后端文档写明）；自测覆盖两种边界之一。
- [ ] 上下文条数/长度限制在服务端生效（与 PRD 中 MAX_MESSAGES_IN_CONTEXT 语义一致或迭代更新说明）。

---

## US-5：前端仅提交本轮用户内容

**作为** 前端实现，**我想要** 每次发送只携带**本轮用户输入**及会话/模型元数据，**以便** 减小请求体并便于后续优化。

### 验收标准（AC）
- [ ] 典型请求体**不包含**完整历史消息数组作为唯一上下文来源；至少包含 `sessionId`、本轮用户文本、`provider`（及智谱所需的 `model`）。
- [ ] 在保持多轮语义正确的前提下，网络面板可观察到请求体大小**不随历史条数明显线性增长**（抽查 2 条 vs 20 条历史对比）。
- [ ] 若保留短期兼容路径，须在 `docs/backend/` 或实现说明中标注废弃计划；本迭代 US 验收以目标态为准。

---

## US-6：数据层可替换

**作为** 维护者，**我想要** 数据访问集中在可替换的抽象之后，**以便** 后续切换到 PostgreSQL 等数据库时改动面可控。

### 验收标准（AC）
- [ ] 业务/API 层不直接散落 `better-sqlite3`（或所用驱动）调用；通过统一接口完成会话与消息的 CRUD。
- [ ] `iterations/0.0.2/backend/`（及同步 `docs/backend/`）中有简短说明：**新增适配器**需实现哪些方法、连接如何注入。
- [ ] SQLite 文件路径可配置或有清晰默认，文档中写明。
