# PRD：多会话对话、SQLite 持久化与请求体瘦身（version=0.0.2）

## 1. 需求摘要

### 1.1 背景
当前对话页在**单次访问**内可多轮聊天，但**仅有一个逻辑对话**；刷新或关闭后历史不持久（见基线 `prd-ai-agent.md` 首版范围）。同时，前端在每次调用对话 API 时**携带当前会话的全部消息**，请求体随轮次增长，不利于带宽、日志与后续多轮优化（如服务端裁剪、工具调用上下文管理）。

本迭代在**本地**引入 **SQLite** 持久化「多个对话会话」，并调整契约：**服务端按会话标识加载历史**、**客户端仅提交本轮新增内容**（在明确约束下）。数据访问须**抽象为可替换实现**，以便后续快速切换至 PostgreSQL 等数据库。

### 1.2 目标用户
通过浏览器使用 AI 对话页的终端用户；需要在多个主题间切换、隔日继续某条对话。

### 1.3 核心场景
1. 用户可**新建对话**，与现有对话并列存在。
2. 用户可从列表（或等价 UI）**切换到历史对话**，继续发送消息；界面展示该会话的完整历史（以服务端/本地存储为准）。
3. 用户发送一条新消息时，请求体**不再重复发送整段历史**；服务端根据**当前会话 ID** 读取已存消息，组装调用模型所需的 `messages` 上下文。
4. 部署在单机/本机场景下，对话数据写入 **SQLite**；更换存储时仅需替换数据层实现，业务与 API 形状尽量稳定。

### 1.4 成功指标
- 至少存在 **N≥2** 个会话时，切换会话后消息互不串线（定性 + 自测用例）。
- 单次对话请求的 JSON 中，**用户输入相关字段**不随历史条数线性膨胀（例如仅含 `sessionId` + 本轮 `content` 或等价最小载荷）。
- 数据层存在清晰 **边界**（接口/模块），文档中说明如何新增另一种 `DatabaseAdapter`（或等价名）而不改 UI 与模型调用核心逻辑。

## 2. 功能范围

### 2.1 做（In Scope）
- **会话模型**：会话具有稳定 **sessionId**（如 UUID）；每条消息归属某一 session，含 role、content、时间序等最小字段（具体字段由设计与后端定稿）。
- **多会话 UI**：支持「新建会话」、展示会话列表或等价导航、**当前会话**高亮或明确指示。
- **持久化**：使用 **SQLite** 存储会话与消息；文件路径可配置（环境变量或约定默认路径），满足本地开发部署。
- **数据层抽象**：所有读写通过统一接口（如 Repository / Port）；SQLite 仅为默认适配器；文档约定扩展点与线程/连接生命周期注意点（Next.js Node runtime）。
- **API/服务行为**：处理聊天请求时，**根据 sessionId 解析当前会话**，从存储读取历史，再合并本轮用户输入，调用现有流式模型链路；流式结束后**将本轮 user/assistant 消息写回**该 session（顺序与幂等策略由实现说明）。
- **请求体调整**：前端提交**本轮用户消息**（及 `sessionId`、`provider`、`model` 等必要元数据），**不**再发送完整 `messages[]` 作为唯一上下文来源（可与后端约定兼容迁移期，见假设）。

### 2.2 不做（Out of Scope）
- **多用户账号与权限**：不要求登录；不区分租户（单机单用户假设即可）。
- **跨设备同步**：不要求云端同步；SQLite 为本地文件。
- **非 SQLite 的第二套存储实现**：本迭代只要求**架构上可插拔**并文档化，不强制交付 PostgreSQL 适配器代码。
- **复杂检索**：全文搜索、按标签筛选会话等可作为后续迭代。

### 2.3 依赖与假设
- 仍基于 Next.js App Router、`/api/chat`（或演进后的路径）与现有 Provider 抽象。
- **兼容策略**（待设计/后端细化）：若短期内需兼容旧客户端，可支持「带全量 messages 且无 sessionId」的旧请求路径；**目标态**为 sessionId + 单条输入。产品验收以目标态为准，兼容为可选。
- 消息条数上限、上下文窗口裁剪（如仅取最近 K 条）延续或扩展 `MAX_MESSAGES_IN_CONTEXT` 语义，**在服务端**执行，与「前端不送全量」一致。
