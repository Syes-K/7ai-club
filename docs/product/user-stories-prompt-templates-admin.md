# 用户故事：提示词模板管理（version=0.0.9）

## US-0.0.9-1：进入提示词管理页

**作为** 管理员  
**我希望** 在控制台中有单独的「提示词管理」入口  
**以便** 不与普通 JSON 配置混在一屏时也能找到并编辑大模型相关模板  

**验收要点**

- 从 console 主导航可进入该页（路径由设计定稿，如 `/console/prompts`）。  
- 页面列出本迭代支持的提示词键及简短说明。

---

## US-0.0.9-2：编辑并保存模板

**作为** 管理员  
**我希望** 每个提示词表单项是**中文标签**，并带有 **tip 说明含义和用法**；用多行文本编辑模板（在允许的键上使用 `{{占位符}}`）  
**以便** 无需读代码或英文键名也能理解每条模板做什么、怎么改  

**验收要点**

- 表单项 **`label` 为中文**（与 `spec-prompt-templates-admin.md` §2.2 一致）；英文配置键不作为主标签。  
- 每条模板具备 **tip**（如 `Form.Item` 的 `tooltip` +/或 `extra`），覆盖**含义**与**用法**，文案不少于设计文档 §2.5 所要求的信息量。  
- 保存成功后在服务端读取路径中生效（与现有配置刷新策略一致）。  
- 非法占位符或空模板时保存失败或明确警告（与 PRD 定稿一致）。  
- 可提供「恢复默认」将模板重置为内置默认。

---

## US-0.0.9-3：服务端统一渲染

**作为** 系统  
**我希望** 摘要注入前缀与摘要系统提示均通过同一模板渲染函数生成  
**以便** 行为一致、易测、易扩展新键  

**验收要点**

- `buildMessagesWithContextSummary` 使用渲染后的前缀 + 摘要正文。  
- `refreshContextSummaryFullRewrite` 使用渲染后的系统提示（含 `maxChars` 等变量）。  
- 配置缺失时使用内置默认，不阻断对话与摘要流程。

---

## US-0.0.9-4：摘要消息识别仍正确

**作为** 开发者  
**我希望** 在自定义前缀后，`messagesIncludeContextSummary`（或替代方案）仍能正确判断首条系统消息是否为摘要注入  

**验收要点**

- 满足 PRD §2.1.1 方案 A 或 B 之一并有单测或文档用例说明。
