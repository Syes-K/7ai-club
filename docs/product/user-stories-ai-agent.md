# AI Agent 用户故事与验收标准

本文档与 `prd-ai-agent.md` 配套，供设计与开发阶段对照验收。故事按「作为…我想要…以便…」格式编写，并附验收标准（AC）。

---

## US-1：选择 AI 模型

作为 **使用浏览器的用户**  
我想要 **在对话前或对话中选择要使用的 AI 模型（DeepSeek 或智谱下某一具体 GLM）**  
以便 **我能按需切换不同模型获得回答**。

**验收标准：**

- [ ] AC1.1 页面上有明确的「模型选择」入口或控件，可区分当前选中的模型（含智谱具体型号）。
- [ ] AC1.2 用户可在 **DeepSeek** 与 **智谱多款 GLM**（如 `glm-4-flash`、`glm-4`、多模态系列等，见 PRD §2.1）之间切换；切换后新发送的消息使用新选中的路由（`provider` + 智谱时 `model`）。
- [ ] AC1.3 切换模型后，已有消息列表中的历史消息仍正常展示，不会被清空或重算。
- [ ] AC1.4 默认预选 **智谱 `glm-4-flash`**（与 `lib/chat/route-key.ts` 一致），用户未操作时可直接发送；也可先改为 DeepSeek 再发送。

## US-2：发送问题并触发请求

作为 **使用浏览器的用户**  
我想要 **在输入框中输入问题并发送**  
以便 **向当前选中的 AI 模型发起对话请求**。

**验收标准：**

- [ ] AC2.1 输入框支持**多行**文本；可通过**发送按钮**或 **Ctrl + Enter**（macOS：**⌘ + Enter**）触发发送；**Enter** 单独按下为换行。
- [ ] AC2.2 发送后，用户输入的内容作为一条「用户消息」立即展示在消息列表中。
- [ ] AC2.3 发送后输入框清空（或可配置清空），并进入「等待 AI 回复」状态（见 US-3 加载状态）。
- [ ] AC2.4 在 AI 正在回复（请求进行中）时，输入框或发送按钮可禁用或限制重复发送，避免重复请求。

---

## US-3：接收 AI 的流式回答

作为 **使用浏览器的用户**  
我想要 **在发送问题后看到 AI 的回复以流式方式逐字或逐段出现**  
以便 **更快感知到回复已开始，并减少长时间白屏等待**。

**验收标准：**

- [ ] AC3.1 用户发送消息后，消息列表中很快出现一条「AI 消息」占位或开始渲染。
- [ ] AC3.2 AI 的回复内容以流式形式逐步展示（逐字或逐段），而非整段一次性出现。
- [ ] AC3.3 流式输出过程中，消息列表自动滚动到底部（或提供「到底部」的体验），便于用户跟随最新内容。
- [ ] AC3.4 流式结束后，该条 AI 消息完整展示，且与用户消息在视觉上可区分（如左右或样式区分）。

## US-4：多轮对话与上下文

作为 **使用浏览器的用户**  
我想要 **在同一会话中连续追问多条消息**  
以便 **AI 能结合之前的对话内容给出连贯回答**。

**验收标准：**

- [ ] AC4.1 用户可在同页面上连续发送多条消息，每条用户消息后对应一条 AI 回复，按时间顺序展示。
- [ ] AC4.2 后端请求时携带当前会话内的历史消息（或由后端维护会话上下文），AI 回复能体现对前文的引用或连贯性。
- [ ] AC4.3 刷新或关闭页面后，对话历史可不保留（首版不持久化）；在同一页面、未刷新前提下，多轮对话始终有效。

---

## US-5：加载状态

作为 **使用浏览器的用户**  
我想要 **在等待 AI 回复时看到明确的加载反馈**  
以便 **我知道请求已发出、系统正在处理，而非卡死或未响应**。

**验收标准：**

- [ ] AC5.1 用户点击发送后，在 AI 回复首字/首段出现前，界面有加载态（如骨架、loading 图标或「AI 正在思考…」等文案）。
- [ ] AC5.2 加载态出现在消息列表中「即将出现 AI 回复」的位置，或与输入区有明确关联，不造成误解。
- [ ] AC5.3 流式首字/首段到达后，加载态消失，由实际流式内容替代。

---

## US-6：错误与重试

作为 **使用浏览器的用户**  
我想要 **在请求失败、超时或服务不可用时看到明确错误提示，并能重试**  
以便 **我理解发生了什么并有机会再次尝试**。

**验收标准：**

- [ ] AC6.1 当请求失败（如网络错误、4xx/5xx、API 限流、模型不可用等）时，界面展示可读的错误提示，不仅依赖控制台或空白。
- [ ] AC6.2 错误提示与「用户消息」或「该轮对话」关联（如在该轮 AI 回复位置展示错误信息），用户能明确知道是哪一次请求出错。
- [ ] AC6.3 在错误状态下，提供「重试」或类似操作，用户点击后可重新请求该问题（或重新发送上一条消息），无需重新输入。
- [ ] AC6.4 若产品提供「清空对话」或「重新开始」，在错误态下也可使用，清空后可开始新对话。

## 故事与待设计项对应关系

| 用户故事 | 主要涉及的待设计项（见 PRD §3） |
|----------|--------------------------------|
| US-1 选择模型 | 模型选择、对话页 |
| US-2 发送问题 | 输入框与发送、对话页、消息列表 |
| US-3 接收流式回答 | 消息列表、加载状态 |
| US-4 多轮对话 | 消息列表、对话页 |
| US-5 加载状态 | 加载状态、消息列表 |
| US-6 错误与重试 | 错误状态、消息列表、输入框与发送 |

设计阶段可根据上表在说明中标注「对应 US-x / ACx.x」，便于前端实现与验收追溯。
