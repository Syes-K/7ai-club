# 用户故事：会话上下文滚动摘要（version=0.0.5）

## 故事与验收标准

### US-1 长对话下的语义延续

**作为** 长时间使用同一对话的用户，**我想要** 在对话轮数超过「模型单次可见条数」后，模型仍能大致记住更早讨论过的要点，**以便** 不必重复粘贴前文。

**验收标准（AC）**

1. 给定一会话，其持久化消息条数大于配置中的「最近 K 条」窗口，且已生成非空摘要；调用对话 API 时，发往模型的消息序列中在「最近 K 条原文」之前存在**一段由系统维护的摘要内容**（角色与精确定义以设计/后端说明为准）；且该摘要的生成须满足 PRD **§2.1.1 摘要更新不变量**（窗口外前缀须被整段重写或合并增量覆盖，禁止仅替换为最新一小段导致中间轮次断层）。
2. 在受控测试脚本中：早期轮次写入唯一令牌/实体名，在远晚于窗口的轮次提问；在开启摘要且摘要已更新的前提下，模型回答应能**引用或识别**该令牌（定性，允许表述不完全一致但不可完全臆造无关内容）。
3. 关闭摘要功能（若配置支持）时，行为与当前版本一致：仅最近 K 条原文进入模型。

---

### US-2 完整历史仍可查阅

**作为** 用户，**我想要** 在侧栏/主区域仍能看到**全部**已保存的 user/assistant 消息，**以便** 核对原文与复制。

**验收标准（AC）**

1. 加载会话消息列表时，返回的条数与摘要功能开启前后**一致**（不因摘要而隐藏早期消息）。
2. 摘要内容**默认不替代**某条用户可见消息；若实现为单独存储，界面不无故减少消息条数。

---

### US-3 摘要失败不阻断聊天

**作为** 用户，**我想要** 在摘要服务暂时失败时仍能正常发消息并得到回复，**以便** 不中断工作。

**验收标准（AC）**

1. 模拟摘要生成失败（如 API 错误）；用户发送消息仍返回 200 与流式正文，行为等价于无摘要时的「仅最近 K 条」。
2. 服务端记录可观测日志（级别与字段由后端说明），便于排查。

---

### US-4 会话删除后无残留

**作为** 用户，**我想要** 删除某会话后，其摘要与消息一并清除，**以便** 无孤立数据。

**验收标准（AC）**

1. 删除会话（现有删除能力）后，该会话 identifier 下不再存在摘要记录（若摘要存于会话行或级联表，由实现保证）。

---

### US-5 可配置的窗口与摘要行为（最小集）

**作为** 部署者/管理员，**我想要** 能调整「最近多少条原文进模型」及摘要相关开关/阈值，**以便** 在成本与效果间平衡。

**验收标准（AC）**

1. 至少支持：**开启/关闭摘要**（或等价：阈值为 0 表示关闭）、**触发摘要更新的最小消息条数差**（或与现网 `maxMessagesInContext` 联动的清晰规则），文档与控制台/配置文件一致。
2. 非法配置被拒绝并返回明确错误（沿用现有控制台校验模式）。
