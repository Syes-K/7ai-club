# PRD：会话上下文滚动摘要（version=0.0.5）

## 1. 需求摘要

### 1.1 背景

多会话与 SQLite 持久化（0.0.2）已保证**界面可展示完整历史**；服务端在调用模型时仍按 `maxMessagesInContext` 仅取**最近 N 条**消息作为上下文。当对话轮数超过 N 时，更早内容不会进入模型请求，用户会感觉「模型忘了前面说过什么」——这是**上下文窗口策略**导致的语义丢失，而非消息从数据库删除。

本迭代采用**方案 2**：在控制 token 的前提下，对**超出最近窗口**的历史做**压缩摘要**，并与「最近 K 条原文」组合后送入模型，使长对话仍能保持**话题与关键事实**的延续性。

### 1.2 目标用户

使用本应用进行多轮、长文本对话的终端用户；需要在数十轮之后仍能依赖模型记住约束、名词与决策。

### 1.3 核心场景

1. 某会话消息条数（或等价度量）超过配置阈值时，系统在后台或请求路径内**维护/更新**该会话的**一条滚动摘要**（文本），摘要概括「窗口外」已发生对话的要点。
2. 每次发起模型补全时，服务端组装的上下文为：**（可选）摘要系统消息 + 最近 K 条 user/assistant 原文**（K 受 `maxMessagesInContext` 或本迭代定义的专用上限约束，二者关系见下文）。
3. 用户仍可在界面查看**完整逐条历史**；摘要仅影响**发往模型的 payload**，不改变既有消息列表的存储与展示（除非产品另定「展示摘要块」的增强，本迭代以服务端行为为主，展示为可选见范围）。
4. 摘要生成失败时，行为**可降级**为仅使用最近 K 条原文（与当前版本一致），并记录日志，不阻断发消息。

### 1.4 成功指标

- 在消息条数明显大于 `maxMessagesInContext` 的测试会话中，模型仍能回答依赖**较早轮次**中明确给出的约束或实体名称（定性验收用例）。
- 单次请求中摘要 + 最近原文的总消息条数/字符数受配置约束，避免无界增长。
- 单机部署下摘要与会话绑定持久化；会话删除时摘要一并清理。

## 2. 功能范围

### 2.1 做（In Scope）

- **会话级滚动摘要**：每个会话最多维护**一段**当前有效的摘要文本（可存于 `chat_sessions` 新列或独立表，由设计/后端定稿）；每次**成功**更新后的摘要须满足 **§2.1.1 不变量**（见下）。
- **摘要更新不变量（§2.1.1，必达）**：设持久化消息总数为 `n`，`K = maxMessagesInContext`，参与模型调用的**原文**为**最近 K 条**。记「窗口外前缀」为按时间顺序的第 `1` 条至第 `n − K` 条（当 `n ≤ K` 时无窗口外内容，摘要可为空或不参与组装）。在**任意**一次摘要刷新成功后，保存的摘要文本 `S` 须**在语义上覆盖当前时刻的全部窗口外前缀**（在「摘要最大字符数」等硬性长度限制下允许有损压缩，但**不得**出现「某段对话已滚出最近 K 条、又未被纳入当前 `S`」的**结构性断层**）。  
  **允许的实现方式**（二选一或等价变体，后端文档写明具体算法）：  
  - **整段重写**：以窗口外全部消息（或其在 DB 中的等价拼接）为输入，调用摘要模型生成新的 `S`；或  
  - **合并增量**：以**旧摘要 + 自上次有效摘要以来新滚出窗口、且尚未被当前 `S` 覆盖的消息文本**为输入，生成新 `S`（须保证合并后仍覆盖完整窗口外前缀）。  
  **禁止**：仅对「本轮新冒出的一小段」单独成文并**替换**为新的 `S`，而不与旧摘要或更早窗口外内容合并/重写，导致中间轮次既不在最近 K 条也不在 `S` 中。
- **组装顺序**：调用模型前，将摘要作为 **一条 `role: system`（或产品约定的单一前置消息）** 置于**最近 K 条原文之前**；若摘要为空则行为与现网一致。
- **与 `maxMessagesInContext` 的关系**（须在设计/后端文档中写死其一，建议默认）：
  - **建议默认**：「最近 K 条」中的 K **等于**现有 `maxMessagesInContext`；摘要单独计数或占用 1 个「虚拟条数」 slot，且总输入不超过模型与运维可接受上限（可在控制台增加「摘要最大字符数」等）。
- **触发条件**（可配置）：例如「持久化消息条数 > K + 缓冲条数」时触发摘要更新，或每 M 轮用户消息触发一次；避免每条消息都调用摘要模型（成本控制）。
- **摘要模型**：优先复用当前会话用户选择的 **provider/model** 或单独配置「摘要用模型」（控制台可扩展，最小实现可用与对话相同模型 + 较短 `max_tokens`）。
- **失败与降级**：摘要 API 失败时跳过摘要，仅用最近 K 条；不打断用户发送与流式回复。
- **隐私与数据**：摘要与消息同属本地 SQLite，适用范围与现有「单机单用户」一致。

### 2.2 不做（Out of Scope）

- **向量检索 / RAG**、跨会话记忆、多用户账号。
- **强制云端存储**；仍以本地 SQLite 为准。
- **自动总结以外的复杂记忆结构**（如结构化 JSON 记忆库），可作为后续迭代。

### 2.3 依赖与假设

- 依赖现有 `ChatStore`、`/api/chat` 流式链路、`getAppConfig()` 配置体系。
- 0.0.4 控制台 JSON 配置可扩展新字段（摘要开关、触发阈值、摘要最大长度等），或与现有 `maxMessagesInContext` 同屏展示。
