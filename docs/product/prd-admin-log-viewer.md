# PRD：管理端日志查看与检索（version=0.0.6）

## 1. 需求摘要

### 1.1 背景

迭代 `0.0.1` 已实现聊天/API 相关结构化日志落盘：按小时切分的行分隔 JSON 文件，路径约定为项目根 `.logs/`（如 `YYYY-MM-DD-HH.log`）。开发者在本地或预发排障时，直接阅读原始文本效率低、难以快速定位单条记录与关联 `requestId`。

本迭代在**不替代**既有写日志能力的前提下，补齐**受控的管理/后端侧能力**：以结构化方式浏览、过滤与检索日志，显著降低开发期排障成本。

### 1.2 目标

- 提供面向开发/排障场景的日志**可读展示**与**便捷检索**（至少支持关键维度过滤与关键词搜索）。
- 在需求层面明确**访问控制与安全边界**（含敏感字段认知），避免日志查看能力成为数据泄露面。
- 在需求层面约定**存储抽象与扩展边界**：当前默认数据源为本地文件；未来可接入其他存储（如数据库、对象存储、日志平台）时，**产品能力描述（浏览、检索、详情）保持稳定**，不因底层存储更换而重新定义用户价值。

### 1.3 与 0.0.1 的关系

- **延续**：继续依赖 `0.0.1` 定义的文件日志格式与字段语义；查看器以「能消费现有行 JSON」为最低兼容目标。
- **扩展**：`0.0.1` 中明确 **Out of Scope** 的「前端页面日志展示与查询」由本迭代纳入（实现形态可为受保护的管理路由 + 服务端读取，具体由设计与开发阶段定稿）。

---

## 2. 目标用户与核心场景

| 角色 | 说明 |
|------|------|
| 后端/全栈开发者 | 实现与维护聊天 API、排查流式与上游错误。 |
| 技术负责人 / 排障协作者 | 需要快速把现象对应到 `requestId` 与日志阶段。 |

**核心场景：**

1. **按次请求追踪**：已知或从错误信息中得到 `requestId`，希望在日志中快速找到该次请求相关条目并查看结构化字段（含 `messages`、响应预览等，以现有落盘字段为准）。
2. **按 level、event、绝对时间区间与关键词缩小范围**：在用户指定的 **绝对时间区间**（明确的起始时刻与结束时刻）内，按 **日志级别（`level`）**、**事件名（`event`）** 精确筛选，并可结合关键字（如模型名、错误片段）进一步缩小候选日志。
3. **单条可读**：将一行 JSON 以人类可读方式展示（字段分组、高亮、折叠等由设计定义），避免纯文本墙。
4. **安全默认**：仅在**受信部署**下暴露 Console（例如本地开发、内网、或网关已鉴权的预发）；**本迭代不设日志专用 API 密钥**；不向匿名公网开放等同「任意读聊天内容」的能力。

---

## 3. 成功指标

1. **可检索**：用户可在合理时间内通过 `requestId`、`level`、`event`、**绝对时间区间**及关键词定位到目标日志条目（具体交互由设计定义，指标以「无需手工翻完整文件」为准）。
2. **可读性**：单条日志的关键字段（时间、级别/事件类型、`requestId`、摘要信息）在默认视图中清晰可见；复杂嵌套结构可展开查看。
3. **访问边界可解释**：文档与界面能说明日志敏感性、**由部署与整站 Console 访问策略**决定谁能到达日志 API；错误响应不静默伪装成功、不扩大泄露面。
4. **扩展不推翻需求**：后续若增加「第二存储后端」，本 PRD 中的用户价值（查看、检索、详情、安全边界）仍成立，仅需在实现层替换或组合数据源，无需重新定义产品目标（见第 6 节）。

---

## 4. 功能范围

### 4.1 In Scope（本迭代）

- **日志消费与展示**：从**当前项目约定的默认数据源**读取日志记录并在管理界面中展示（MVP 默认与 `0.0.1` 一致：**项目下的按小时 JSONL 文件**）。
- **检索与过滤**（至少覆盖以下能力的产品定义，具体控件与查询语法由设计/实现细化）：
  - 按 **时间（绝对时间区间，必选语义）**：用户指定 **明确的起始时刻与结束时刻**（日期 + 时间，精度至少到**分钟**；是否支持秒级由实现决定并文档化）。**过滤以每行 JSON 的 `ts` 字段为准**：只展示 `ts` 落在该区间内的记录；区间边界开闭（建议统一为**闭区间** `[start, end]` 或**左闭右开** `[start, end)`）须在实现与 API 文档中写明。按小时日志文件遍历仅为读取优化，**不得**把产品上的「时间筛选」降级为「仅选择某个小时文件」作为唯一方式。
  - 按 **`level`**：与现有 JSON 行内 `level` 字段对齐（如 `info`、`warn`、`error` 等）；支持单选或多选（以实现为准），无该字段的行须有明确展示或过滤规则（如归入「未知」或始终出现在「不限 level」结果中——实现须在文档中写明）。
  - 按 **`event`**：与现有 JSON 行内 `event` 字段对齐；支持从可选列表选择、和/或自由输入事件名（以实现为准，需避免误拼时零反馈）；支持多 event 时明确为 OR 还是 AND。
  - 按 **`requestId`** 精确或前缀匹配（以实现能力为准，需在故事中验收）。
  - 按 **关键词** 在整行或选定字段中搜索（字段范围可在实现中定义并文档化）；与 `level`/`event`/绝对时间区间组合时语义为**交集**（除非文档另有说明）。
- **单条详情**：从列表进入单条记录的详情视图，或等价的信息层次，保证结构化阅读体验。
- **访问控制（部署侧）**：必须通过**网络/网关/整站 Console 策略**限制谁能访问管理端与相关 API；**本迭代不要求**在日志 API 上增设独立环境密钥或请求头校验（与 `api-spec-admin-log-viewer` 一致）；部署方须避免将 Console 暴露给不可信公网。
- **实现层扩展预留（需求表述）**：在架构上应能通过**存储抽象**接入非文件后端；本迭代**不要求**完成第二种存储实现，但需求上不得把「只能读文件」写死为永久产品约束（见第 6 节）。

### 4.2 Out of Scope（本迭代不做）

- 第三方 APM/日志平台（如 ELK、Datadog、云厂商日志服务）的**正式**接入与运维级仪表盘。
- **中心化审计日志**、合规留存周期、跨地域多副本——本能力定位为**开发/排障辅助**，不承诺长期审计存储。
- 复杂的组织级 RBAC（多租户、细粒度字段级权限）；本迭代以「小范围受信部署 + 网关/内网隔离」为主。
- 日志**写入路径、字段 schema、文件命名规则**的破坏性变更（若需变更应另开迭代并与 `0.0.1` 文档对齐）。
- 实时流式「tail -f」级订阅（可作为后续增强，本迭代不强制）。

### 4.3 依赖与假设

- **依赖**：`0.0.1` 日志文件存在且格式为**每行一个 JSON 对象**；应用进程对项目根 `.logs/` 下文件可读；**绝对时间区间筛选**依赖行内可解析的 **`ts`**（与现有落盘一致，一般为 ISO 8601）。缺失或无法解析 `ts` 的行的处理规则（排除 / 单独归类等）须在实现文档中说明。
- **假设**：主要使用者为受信开发者；生产环境若启用，必须叠加额外网络与身份措施（由部署方负责，产品侧在风险中声明）。

## 5. 风险与约束

### 5.1 敏感信息

- 日志中可能包含 **用户消息内容、模型回复预览、业务上下文**，属于高敏感数据。
- **约束**：查看能力不得在无边界公网场景默认可达；部署方须限制 Console 暴露面；禁止依赖「仅前端隐藏 URL」作为唯一防护。
- 建议在实现中提供**字段截断/脱敏开关**或展示层默认折叠长文本（具体方案设计阶段定，本 PRD 要求考虑并在验收中体现「降低误泄露概率」的措施之一）。

### 5.2 访问控制与安全

- **风险**：管理端成为横向移动与数据泄露入口。
- **约束**：
  - 默认由**部署**决定可达范围（内网、VPN、网关鉴权等）；**本迭代**日志 API 不设专用密钥门禁。
  - 不得依赖仅前端隐藏路由的「假安全」；若整站日后增加统一鉴权，日志接口随整站策略即可。
  - 错误信息不得泄露服务器路径或内部实现细节（在故事与验收中体现）。

### 5.3 运行环境与存储

- `.logs` 可被人工删除或受磁盘/权限影响不可用；本查看器**不承诺**恢复已删除文件。
- 大文件全量扫描可能影响性能；实现上应考虑**分页、按文件倒序、限制扫描深度或时间窗口**等（由设计/后端细化，需求侧要求「在常规开发数据量下可用」）。

### 5.4 合规与责任边界

- 本功能**不**单独满足医疗/金融等强合规场景的审计要求；若未来有合规需求，应另立迭代与存储策略。

## 6. 未来「多存储后端」的边界说明（扩展性）

### 6.1 产品能力层（稳定）

以下用户能力在更换存储后端时应保持不变：

- 浏览日志条目列表（带分页或等价机制）。
- 按 **绝对时间区间**、`level`、`event`、`requestId`、关键词等维度检索。
- 查看单条结构化详情。
- 在受控部署下访问（访问模型可随整站鉴权/网关增强；产品能力层仍要求**不**将高敏感日志能力设计为「默认可对任意公网匿名开放」）。

### 6.2 实现层（可替换）

- **数据源**：MVP 为「文件型 JSONL」；未来可替换为「查询数据库」「调用内部日志 API」「对象存储 + 索引」等，应在实现上通过**存储抽象**（例如统一的 list/query/get 语义）对接，避免 UI 与业务逻辑硬编码文件路径遍历。
- **本迭代交付边界**：**不强制**引入具体数据库或消息队列；若 MVP 仅实现文件后端，只要在架构与文档上承认后续可插拔第二实现即可。若团队判断「无抽象会导致必然重写」，则允许在本迭代实现最小抽象接口，但**不作为**对外产品功能承诺（仍属工程内部质量）。

### 6.3 与「不锁定文件」的表述方式

- 需求文档：**默认数据源 = 当前仓库约定的文件日志**；**允许且预期**存在其他后端实现，产品描述以「日志查看与检索服务」为中心，而非「文件浏览器」为永久名称或唯一隐喻。

| D-05 | 状态 | **空状态、加载中、错误**（目录不可读、搜索无结果等）的文案与视觉；**不含**日志专用 API 403/密钥未配置专页。 |

| D-07 | 响应式 | 以桌面优先；若窄屏仅保证「可用」，不强制完整大屏信息密度。 |
