# AI Agent 对话产品需求文档（PRD）

## 1. 需求摘要

### 1.1 背景
需要开发一个 **AI Agent**，让用户通过**浏览器**与 AI 进行对话、获取 AI 的回答。首期支持智谱（Zhipu/GLM）与 DeepSeek 两家模型；技术架构上需便于后续扩展更多模型/厂商。

### 1.2 目标用户
- **主要用户**：通过浏览器访问产品的终端用户，希望快速向 AI 提问并得到回答。
- **使用场景**：日常问答、知识查询、简单任务辅助（如文案润色、摘要等），单次访问内完成多轮对话。

### 1.3 核心场景
1. 用户打开页面，选择要使用的 AI 模型（智谱或 DeepSeek）。
2. 用户输入问题并发送，界面展示加载状态。
3. 用户以**流式**方式看到 AI 回复逐字/逐段出现。
4. 用户可在同一会话中继续追问，AI 基于上下文回答。
5. 出现错误（如网络、API 限流、模型不可用）时，界面给出明确提示并可重试。

### 1.4 成功指标
- 用户能完成「选择模型 → 发送问题 → 收到流式回答」的完整闭环。
- 智谱、DeepSeek 两家均可正常调用，且切换模型后对话行为符合预期。
- 后端模型调用层抽象清晰，新增一家模型时改动范围可控（可扩展性）。
- 错误与加载状态有明确反馈，用户可理解并执行重试等操作。

## 2. 功能范围

### 2.1 做（In Scope）
- **对话界面**：单页或明确主流程页面，包含消息列表、输入框、发送与模型选择入口。
- **模型支持**：首期支持**智谱（Zhipu/GLM）**与 **DeepSeek**；智谱侧可在 **GLM-4-Flash、GLM-4、GLM-4.6V 系列（含 Flash / FlashX）** 等具体模型间切换（下拉文案区分各版本特点）；前端可切换模型，后端统一抽象（如 Provider/Adapter），便于扩展。
- **流式输出**：AI 回复以流式（SSE 或类似）返回，前端逐段渲染，而非整段一次性展示。
- **多轮对话**：同一会话内保留对话历史，请求时携带上下文（条数或长度可配置），保证多轮语义连贯。
- **模型选择**：用户可在对话前或对话中切换当前使用的模型；切换后新消息使用新模型，旧消息仅展示不重算。
- **加载与错误状态**：请求中显示加载态；请求失败或超时时展示错误信息，并提供重试或清空等操作。
- **技术栈**：后端与前端均使用 **Next.js + TypeScript**，与现有 WORKFLOW 约定一致。

### 2.2 不做（Out of Scope，首版）
- **登录/注册**：首版不要求用户登录，匿名使用即可；若后续需要「我的对话」再考虑账号体系。
- **持久化对话历史**：首版不落库、不跨会话持久化；刷新或关闭页面后历史清空，仅当次会话有效。
- **对话管理**：不提供「保存对话」「重命名会话」「会话列表」等；仅单会话、当页有效。
- **文件/图片输入**：首版仅支持文本输入，不支持上传文件或图片作为上下文。
- **模型参数配置**：首版不暴露 temperature、max_tokens 等参数给终端用户，使用服务端默认配置即可。

### 2.3 依赖与假设
- **API Key 与计费**：智谱、DeepSeek 的 API Key 由服务端配置或环境变量管理，不暴露给前端；计费与用量由后端/运维侧处理。
- **Next.js 项目**：已有或将新建 Next.js + TypeScript 项目，API 路由（Route Handlers）与前端页面均在该项目中实现。
- **流式协议**：采用 SSE（Server-Sent Events）或兼容的流式响应，前端通过 fetch/EventSource 或等价方式消费。
- **上下文窗口**：多轮上下文条数或 token 上限由后端配置，前端仅按「当前会话消息列表」发送；具体裁剪策略由后端实现。

## 3. 待设计项（Design Handoff）

以下页面/流程/组件需要设计介入，便于 design 阶段按项拆解交互与视觉。

| 序号 | 类型 | 名称 | 说明 |
|------|------|------|------|
| 1 | 页面 | 对话页（主页面） | 用户进入后的主界面：消息列表 + 输入区 + 模型选择入口；需考虑空态、仅用户消息、已有对话等状态。 |
| 2 | 组件/区域 | 消息列表 | 展示用户与 AI 的消息流；需区分用户消息与 AI 消息的展示样式；支持流式逐字/逐段渲染、滚动与自动滚动到底部。 |
| 3 | 组件 | 输入框与发送 | **多行**文本输入；**Ctrl / ⌘ + Enter** 发送、**Enter** 换行（见 PRD §4）；发送按钮、占位与请求中禁用态。 |
| 4 | 组件/入口 | 模型选择 | **下拉分组**：DeepSeek 与多款智谱 GLM（选项文案区分版本特点，见 `lib/chat/zhipu-models.ts`）；当前选中态、切换后新消息用新路由。 |
| 5 | 状态 | 加载状态 | 用户发送后、等待 AI 回复期间的加载态（如气泡骨架、loading 图标、文案「AI 正在思考…」等）。 |
| 6 | 状态 | 错误状态 | 请求失败、超时、限流等错误时的提示样式与文案；重试、清空或关闭等操作。 |
| 7 | 可选 | 设置/关于 | 若首版需要展示「当前模型」「版本号」或简单设置入口，可单独列出；否则可合并在对话页内。 |

设计说明中请标注与「用户故事/AC」的对应关系，便于前端实现与验收。

## 4. 迭代记录（优化）

- **UI**：对话页增加图标（标题区、消息头像、发送/清空/错误等），空态更友好。
- **输入**：多行文本输入；**Ctrl + Enter**（macOS 为 **⌘ + Enter**）发送，**Enter** 换行。
- **智谱模型粒度**：除 DeepSeek 外，智谱可在多个 GLM 模型 ID 间选择，展示文案说明各版本定位（速度/旗舰/多模态等）。
