# 用户故事：管理端日志查看与检索（version=0.0.6）

本文档与 `prd-admin-log-viewer.md` 配套，故事 ID 前缀 `LOG-` 用于追溯与排期。

---

## LOG-01 浏览日志列表

**作为** 开发者  
**我想要** 在受保护的管理界面中按时间顺序浏览聊天/API 结构化日志条目列表  
**以便** 无需打开 `.logs/` 下的原始文件即可掌握最近发生了什么

**验收标准：**

- [ ] 列表展示来自默认数据源（当前迭代：与 `0.0.1` 一致的 JSONL 文件）的日志条目，至少包含可区分条目的时间、`level`、`event`（或等价摘要列）、`requestId`（若某条缺失字段则明确展示占位或「未知」而非崩溃）。
- [ ] 支持分页、虚拟滚动或「加载更多」之一，避免一次性渲染超大结果集导致页面不可用（具体方案由设计/实现定，需可验收）。
- [ ] 列表数据来源为服务端接口，不依赖浏览器直接读取本地文件路径。

---

## LOG-02 按 requestId 定位

**作为** 开发者  
**我想要** 通过 `requestId` 快速筛选或跳转到相关日志条目  
**以便** 将一次请求在 API 与上下游日志中串联起来

**验收标准：**

- [ ] 提供 `requestId` 输入或筛选能力；输入合法 ID 后，列表仅展示匹配条目或明确提示无结果。
- [ ] 若实现支持前缀/模糊匹配，需在界面或帮助文案中说明匹配规则；若仅支持精确匹配，同样需明示。
- [ ] 从列表可进入该条目的详情视图（或等价信息层级），且详情中包含完整 `requestId`。

---

## LOG-03 按 level、event、绝对时间区间与关键词缩小范围

**作为** 开发者  
**我想要** 按 **`level`**、**`event`**、**绝对时间区间**（起止时刻）筛选，并可结合关键词搜索  
**以便** 在大量日志中快速缩小到可疑记录（例如只看 `error` 或只看 `api.stream_failed`）

**验收标准：**

- [ ] 支持指定**绝对时间区间**：须提供**起始**与**结束**时刻（日期+时间，至少精确到分钟）；筛选结果仅包含行内 **`ts`** 落在该区间内的记录；区间边界开闭与**时区**（本地或 UTC 等）在界面或文档中可发现且与实现一致。不得以「仅选择按小时日志文件名」作为唯一时间筛选方式。
- [ ] 支持按 **`level`** 筛选：选项与当前落盘 JSON 中 `level` 取值一致（可多选或单选以实现为准）；界面或文档说明缺失 `level` 字段时的行为。
- [ ] 支持按 **`event`** 筛选：可与落盘 JSON 中 `event` 取值对齐（建议提供可选列表或自动完成，避免仅靠硬记字符串）；支持多 event 时说明为 OR 或 AND。
- [ ] 关键词搜索对用户可见地说明搜索范围（例如整行文本或指定字段）；无结果时展示空状态而非空白页无解释。
- [ ] **绝对时间区间、`level`、`event`、关键词、`requestId`（若同时填写）组合过滤语义为交集**，除非产品文档另有约定并在界面可发现。
- [ ] 筛选条件变更后，结果与 URL 或内部状态一致（刷新或可分享的具体策略由设计/实现选做，不强制 URL 同步，但行为需可预测）。

## LOG-04 单条结构化可读详情

**作为** 开发者  
**我想要** 以结构化、可展开的方式查看单条 JSON 日志的全部字段  
**以便** 理解事件类型、参数与错误信息，而不必自行格式化原始 JSON

**验收标准：**

- [ ] 详情视图展示该条目的完整结构化内容（与源 JSON 字段一致，允许排序与分组 UI 差异）。
- [ ] 对长文本字段（如 `messages`、响应预览类字段）有默认折叠、截断或标签页等降低信息淹没的处理（至少一种）。
- [ ] 详情从列表进入路径清晰，可返回列表且保留列表筛选状态（若技术限制无法保留，需在界面提示）。

---

## LOG-05 安全默认与访问边界说明

**作为** 安全责任人 / 开发者  
**我想要** 明确日志含敏感数据、访问边界由部署保障，且错误响应不扩大泄露面  
**以便** 在**不设日志专用 API 密钥**的前提下仍降低误暴露风险

**验收标准：**

- [ ] **本迭代**：日志列表/汇总 API **不设**独立密钥或环境开关门禁；谁能访问与整站 **Console**（`/console`）及部署侧网络/网关策略一致（与 `api-spec-admin-log-viewer` 一致）。
- [ ] 列表或错误场景中**不**泄露敏感服务器绝对路径、栈信息或内部实现细节（与现有 API 错误约定一致）。
- [ ] 界面（至少 **Console 琥珀色安全提示条**）与/或文档说明：日志可能含聊天与模型相关内容、勿将管理端暴露公网、建议内网或网关鉴权。

## LOG-06 存储后端可扩展（工程与产品边界）

**作为** 技术维护者  
**我想要** 日志读取逻辑不永久绑定「仅遍历本地文件」一种实现  
**以便** 未来可接入数据库或其他存储而不推翻「列表/检索/详情」产品能力

**验收标准：**

- [ ] 架构或代码层面存在可替换的数据访问边界（如 repository / provider 接口），文件实现仅为其一；若本迭代仅交付文件实现，接口或模块边界仍被定义并文档化（写入 backend 文档，本故事验收以「可追溯的抽象存在」为准）。
- [ ] PRD 中「多存储后端」边界与本文档一致，无「仅支持文件」的永久性产品表述矛盾。
- [ ] 不强制本迭代引入具体数据库或云服务；若未引入第二种实现，不视为故事失败，但抽象与扩展说明必须完备。
