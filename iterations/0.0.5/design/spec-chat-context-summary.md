# 设计说明：会话上下文滚动摘要（version=0.0.5）

**输入需求**：`iterations/0.0.5/product/prd-chat-context-summary.md`、`user-stories-chat-context-summary.md`  
**基线对话**：`docs/design/spec-chat-sessions-sqlite.md`、`docs/design/spec-ai-agent.md`

## 1. 设计决策（PRD D1～D4 定稿）

### D1 信息架构：摘要是否对用户可见

| 决策 | 内容 |
|------|------|
| **本迭代** | **不在对话区增加「摘要」或「对话记忆」展示块**；消息列表仍**仅展示**持久化的 user/assistant（及既有 system，若有）消息，与 0.0.2 一致。 |
| **原因** | 满足 PRD「服务端行为为主」；避免与气泡列表混淆、减少实现与测试面。 |
| **后续可选** | 若产品追加：在**主区顶栏下方**或**第一条消息上方**增加可折叠 **`对话记忆（模型用）`**，展示只读摘要文本；**本迭代不实现**。 |

### D2 状态：摘要生成中

| 决策 | 内容 |
|------|------|
| **本迭代** | **不**增加「摘要生成中…」全局遮罩或禁用输入；用户发消息流程与现网一致。 |
| **原因** | PRD 要求失败降级、不打断；摘要调用应控制在可接受延迟内（实现侧：异步于流式主链路之外，或短超时），前端**无新状态机**。 |
| **例外** | 若实测首字节延迟显著，由后端/产品后续迭代再议 **非阻塞轻提示**（如顶栏小号文案），**不在 0.0.5 必做**。 |

### D3 控制台：新配置项布局与校验

在 **`/console`** 现有 **「模型与上下文」** 卡片内、`maxMessagesInContext` **下方**追加一组字段（同一卡片、同一视觉层级），避免用户误以为摘要条数与「最大上下文消息条数」是同一概念：

| 字段（建议键名，后端可微调） | 控件 | 约束与说明 |
|------------------------------|------|------------|
| **启用上下文摘要** | `checkbox` | 默认 `false` 或 `true` 由后端默认值定稿；关闭时行为与当前版本完全一致（仅最近 K 条进模型）。 |
| **摘要最大字符数** | `number` | 建议范围如 **200～8000**（具体上下限由后端校验与 PRD 对齐）；超出则截断或拒绝保存（后端定稿一种）。 |
| **摘要更新间隔（轮）** | `number` | 表示「每累计 **N 轮用户侧有效轮次**（或每 N 次 assistant 回复后）尝试刷新摘要」或「距离上次摘要后新增消息达到阈值再刷新」——**精确语义见后端 `api-spec`**；前端展示标签：**「摘要更新间隔（轮）」** + 辅助说明：`控制摘要调用频率，数值越大成本越低、摘要越滞后。` |

**文案区分（置于 `maxMessagesInContext` 说明旁或卡片顶注）**：

- **最大上下文消息条数**：参与模型调用的**最近若干条完整对话**（user/assistant）条数上限。
- **上下文摘要**：对**更早**内容的压缩文本，**不**替代消息列表中的气泡；**仅**影响发往模型的上下文。

可选：在「启用上下文摘要」下增加一行 **`text-xs text-zinc-500`**：**「对话列表仍显示全部历史；摘要仅帮助模型回忆更早内容。」**

### D4 降级与错误（用户可见性）

| 场景 | 行为 |
|------|------|
| 摘要 API 失败 / 超时 | **无**对话区错误条；对话正常流式返回。 |
| 配置保存失败 | 沿用 0.0.4：控制台红色错误条，与摘要功能无关的通用保存逻辑。 |

开发者侧日志由后端实现说明；**不对终端用户展示「摘要失败」**。

## 2. 服务端组装顺序（设计约束，供后端实现）

以下**非 UI**，但设计文档固定产品与验收口径：

1. 取会话**滚动摘要**文本 `S`（若未生成或为空，则 `S` 无效）。
2. 取持久化消息列表，截取 **最近 K 条** 转为 `ChatMessage[]`，其中 **K = `maxMessagesInContext`**（与现网一致）。
3. 若 `S` 有效且启用摘要：最终发往模型的序列为 **`[{ role: "system", content: "<前缀>…" }]` + 最近 K 条**；`<前缀>` 建议为固定中文模板，例如：`以下是本对话更早内容的摘要，供你理解上下文（用户消息列表中仍有完整原文可供其查阅）：\n`，后接 `S`。
4. 若摘要关闭或 `S` 无效：仅最近 K 条，**与现网 byte-for-byte 行为一致**（除新增配置读取外）。

**与 PRD 一致**：摘要不占「K 条」内的 user/assistant 计数；**system 摘要单独一条**，需计入 provider 总 token，故需 **摘要最大字符数** 上限。

## 3. 摘要更新：不变量与实现方式（后端约束）

与 PRD **§2.1.1** 对齐，设计侧固定以下口径，避免「更新间隔」与「仅总结最新一段」组合时出现**中间段丢失**：

| 不变量 | 说明 |
|--------|------|
| **覆盖完整窗口外前缀** | 设 `n` 为当前会话持久化消息条数，`K = maxMessagesInContext`。窗口外为第 `1…(n−K)` 条（`n≤K` 时可无摘要）。每次摘要**成功落库**后的 `S`，须代表**当时**全部窗口外内容（受摘要最大字符数限制时允许细节有损，但不得出现整段轮次「既不在最近 K 条也不在 `S`」的断层）。 |

| 允许做法 | 说明 |
|----------|------|
| **整段重写** | 每次触发刷新时，以「当前窗口外全部消息」的文本（从存储读取并拼接）为摘要模型输入，输出新 `S` 覆盖该前缀。 |
| **合并增量** | 输入为 **旧 `S` + 尚未被纳入 `S` 的窗口外新消息**（或等价：旧 `S` + 自上次摘要以来滚出窗口的增量），输出新 `S`；须证明/实现上保证与「整段重写」在覆盖范围上等价，无中间空洞。 |

| 禁止 | 说明 |
|------|------|
| **仅替换为「最新一小段」** | 仅用本轮新溢出片段生成新 `S` 并丢弃旧 `S` 中仍需要的早期要点，导致第 `24` 轮等**中间段**在两次刷新之间既滚出 K 又未进入 `S`。 |

**与「摘要更新间隔」的关系**：间隔只控制**何时调用**摘要 API，**不改变**上述不变量；间隔过长时，在两次刷新之间仍可能短暂存在**旧 `S` 相对当前 `n` 略滞后**的问题，但一旦执行刷新，必须满足完整前缀覆盖（在长度限制内）。

## 4. 用户故事映射

| 故事 | 设计落点 |
|------|----------|
| US-1 | 服务端组装顺序（§2）+ 摘要更新不变量（§3）；控制台可关闭摘要做对照验收。 |
| US-2 | D1：列表仍仅展示存储消息，不插入摘要气泡。 |
| US-3 | D2、D4：无新阻塞 UI、无失败提示。 |
| US-4 | 无 UI；删除会话时摘要随会话删（后端/数据模型）。 |
| US-5 | D3：控制台三项 + 校验说明。 |

## 5. 对话页（ChatApp）变更范围（设计约束）

| 项目 | 说明 |
|------|------|
| **最小变更** | 若公共配置增加字段：`GET /api/config/public` 可**不**向浏览器暴露摘要全文（摘要仅存服务端）；仅当未来展示「对话记忆」时才需前端拉取摘要。本迭代 **ChatApp 可不新增 fetch 摘要**。 |
| **可选** | 若产品希望对话页显示一行说明：「长对话已启用摘要辅助模型记忆」——可作为 **静态说明** 挂在 `maxMessagesInContext` 相关提示旁；**默认不做**，避免干扰。 |
