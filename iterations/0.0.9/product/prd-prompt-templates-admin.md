# PRD：提示词模板管理与服务端组装（version=0.0.9）

## 1. 需求摘要

### 1.1 背景

与大模型交互时，部分文案在服务端以**硬编码**维护（例如 `src/lib/chat/context-summary.ts` 中的 `CONTEXT_SUMMARY_MODEL_PREFIX`、`buildContextSummarySystemPrompt(maxChars)` 等）。调整语气、长度说明或前缀文案需要改代码与发版。希望将这类内容**配置化**：在**管理后台**维护**字符串模板**，服务端按**统一占位符规则**注入运行时计算值后，再参与请求组装。

### 1.2 目标用户

部署本应用的**管理员/开发者**（与现有 `/console` 场景一致）；不要求多租户。

### 1.3 核心场景

1. 访问管理后台中的**独立「提示词管理」页面**（可与现有控制台配置并列导航），查看当前各**提示词键**对应的模板正文。
2. 编辑模板并保存；持久化使用**独立文件** **`data/prompt-templates.json`**（可选用环境变量指定路径），与 `app-config.json` **分离**；读写方式与现有 JSON 配置（原子写、合并默认值）**对齐**。
3. 服务端在组装摘要系统消息、摘要任务系统提示等路径上，**不再**写死长字符串，而是：**读取模板 → 用统一引擎替换占位符 → 得到最终字符串**。
4. 模板缺失、损坏或占位符不合法时，**回退**到内置默认模板（与当前硬编码语义等价），并可在管理端或日志中提示（粒度由实现定）。

### 1.4 成功指标

- 修改「摘要注入前缀」或「摘要系统提示」模板并保存后，**无需改代码**，下一轮对话/摘要刷新即使用新文案（在配置已生效的前提下）。
- 所有本迭代纳入的提示词均通过**同一套占位符与渲染规则**组装；新增键时扩展配置与文档即可，避免多种 ad-hoc 拼接风格。
- `messagesIncludeContextSummary` 等依赖「前缀可识别」的逻辑，在可配置前缀下仍**行为正确**（或与设计约定的新识别方式一致，见 §2.1.1）。

## 2. 功能范围

### 2.1 做（In Scope）

- **提示词键（首版建议最小集）**  
  - **上下文摘要 · 注入模型时的系统消息前缀**（现 `CONTEXT_SUMMARY_MODEL_PREFIX`）：最终注入形式为「渲染后的前缀模板 + 摘要正文」；摘要正文本身**不是**模板的一部分，仍为运行时数据拼接。  
  - **上下文摘要 · 摘要任务系统提示**（现 `buildContextSummarySystemPrompt`）：模板内通过占位符引用 **`maxChars`**（及未来同类标量），服务端渲染后作为摘要调用的 `role: system` 内容。

- **统一字符串模板**  
  - 占位符语法由设计/后端**写死一种**（例如 `{{maxChars}}`、`{{variableName}}`），文档中列出**每个键**允许的占位符及含义。  
  - **禁止**在管理端执行任意代码；仅做**字面量替换**（及明确的换行/转义规则，若有）。  
  - 「函数生成的」与「计算的」差异统一为：**代码侧只提供占位符键 → 值表**，渲染引擎只做替换，不再在业务处散落 `` `${...}` `` 拼接长文案。

- **管理后台**  
  - **独立路由/菜单项**：「提示词管理」或等价名称；每个模板项使用**中文 `label`**（不写死英文配置键作为主标签）；提供 **tip**（悬停/辅助说明）阐述该提示词的**含义**（在请求链路中的作用）与**用法**（如何修改、与占位符及其它配置的关系），详见设计规格 §2.5、§5.4。  
  - **多行文本编辑区**、保存/重置为默认（可选）；保存需经服务端校验：未知占位符是否拒绝或警告（由设计定稿）；空模板是否拒绝（建议拒绝并提示）。

- **服务端**  
  - 在 `buildMessagesWithContextSummary`、`refreshContextSummaryFullRewrite`（及任何依赖上述文案的路径）中改为使用**渲染后的字符串**。  
  - 读取 **`prompt-templates.json`** 经专用模块合并默认；**缓存/失效策略**与控制台保存行为一致（保存后立即生效）；**不**要求并入 `AppConfig`。

#### 2.1.1 识别前缀与可配置前缀（必达其一）

当前实现通过 `messages[0].content.startsWith(CONTEXT_SUMMARY_MODEL_PREFIX)` 判断首条系统消息是否为「摘要注入」。前缀改为可配置后，须满足以下之一并在后端文档写明：

- **A（推荐）**：识别逻辑使用**当前生效的渲染后前缀**（与注入时使用同一模板、同一变量，且摘要正文拼接方式固定），保证 `startsWith` 仍可靠；或  
- **B**：改为不依赖前缀字符串的结构化标记（例如扩展消息元数据，若协议允许）；本迭代若选 B 须评估对存储/API 的影响。

### 2.2 不做（Out of Scope，首版）

- 在管理端维护**用户可见对话页**的每条系统提示（除非与上述键重合）；本迭代聚焦**已识别的服务端组装提示词**。  
- **多语言 UI** 与按 locale 分模板（可后续迭代）。  
- **版本历史、审计、双人审批**、按环境分库。  
- 提示词内的 **A/B 实验** 与流量分割。

### 2.3 依赖与假设

- 依赖现有 Next.js、`/console`、JSON（或等价）应用配置持久化与校验模式。  
- 依赖 0.0.5 上下文摘要行为与数据模型；本迭代**不改变**摘要算法不变量，仅改变文案来源与组装方式。
